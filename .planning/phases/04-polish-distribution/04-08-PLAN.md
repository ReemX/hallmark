<objective>
Fix UAT test 4 root cause #1: repeat tray "Fire test popup" clicks past the 10s
CrossSourceDedup TTL produce no popup because the SQLite UNIQUE INDEX
`idx_unlock_dedup ON unlock_history(app_id, ach_api_name, session_id)` is a
permanent (per-session) second dedup layer. The synthetic test fire always
re-uses the (480, "HALLMARK_TEST_UNLOCK") pair, so every fire after the first
collides on the index → `record_unlock` returns Ok(false) → run_pipeline
silently `continue`s without UNLOCK or POPUP_FIRED logs.

Per the diagnosis (`.planning/debug/popup-repeat-fire-stuck.md` § Resolution),
the recommended Option-1 fix is timestamp-suffixing the synthetic api_name in
`test_trigger::fire` so each click produces a UNIQUE triplet. The popup_queue
gets a small fallback so the user-visible popup still says "Test Achievement"
even though the underlying api_name now varies per fire (matches the locked
UI-SPEC § "Test popup fixture copy" contract).

Crucially, this does NOT alter record_unlock, the UNIQUE INDEX, or production
dedup behavior for real achievements (where re-firing the same achievement IN
ONE SESSION is correctly suppressed). It only changes the synthetic injector
to stop colliding with itself.

Out of scope (deferred / separate task per the diagnosis): promoting the two
silent `tracing::debug!` drop arms in `run_pipeline` to `info!` for
observability. The diagnosis flagged this as a "secondary observability fix" —
not required to close UAT test 4 / unblock UAT test 5.

Output: One backend Rust patch in `test_trigger.rs` (new constants + per-call
timestamp suffix + adjusted unit test), one small substitution in
`popup_queue.rs::process_event` to preserve user-visible copy, one new unit
test verifying the fallback substitution.
</objective>
