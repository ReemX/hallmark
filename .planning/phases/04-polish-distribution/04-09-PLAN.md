<objective>
Fix UAT test 4 root cause #2 + test 14 root cause #1: the "20-second blank
WebView" warmup gap and the structural "SFX plays but popup never paints"
race. Per the diagnosis (`.planning/debug/webview-warmup-blank-screen.md`),
two compounding mechanisms produce the symptom; this plan addresses BOTH:

**Mechanism A (DEV ONLY): Vite multi-entry cold-transform bottleneck.**
vite.config.ts declares 4 rollup inputs (companion/popup/settings/wizard)
but `optimizeDeps` is left implicit. esbuild auto-discovery only walks
index.html, so popup.html / wizard.html / settings.html each pay a full
cold-bundle cost on first GET (10-30s). Fix: explicit `optimizeDeps.entries`
listing all 4 HTML entries plus `optimizeDeps.include` listing the heavy
shared deps. Expected dev warmup: 20s drops to 2-5s.

**Mechanism B (DEV + PROD): Missing WebView-ready handshake.**
popup_queue::process_event calls `emit_to("popup", "popup-show", payload)`
with no await for a "frontend-mounted" ack. Tauri events do NOT buffer for
late listeners. If process_event runs before main-popup.tsx useEffect
registers `listen('popup-show')`, the event is silently dropped. Audio plays
because rodio is independent of WebView state — exactly the "SFX without
popup" pattern. In prod the race window is ~100-500ms (mostly imperceptible);
mechanism A widens it to 20s+ in dev which makes B reliably reproducible.
Fix: a `popup_ready` Tauri command + `tokio::sync::Notify` handle on
AppState; popup_queue::run blocks the first emit on `notify.notified()`. Same
pattern (instrumentation only) for `wizard_ready` and `settings_ready`.

A 5-second timeout backstop guards against frontend mount failures so the
backend never deadlocks. Per RESEARCH § Pitfall 5 (timeout-or-bail pattern),
the timeout fires the first event anyway with `tracing::warn!` so
observability is preserved.

Out of scope:
- Switching UAT signoff to production-build runs (Tier 1 of the diagnosis
  workaround). The fix here is the source-level Tier 2; UAT continues to use
  `cargo tauri dev`.
- Chromium teardown ERROR_CLASS_DOES_NOT_EXIST = 1412 — explicitly deferred

Output: One frontend config change (vite.config.ts), one backend module

       /// Wait on a Notify with a timeout. Returns true if signalled, false on timeout.
       /// Used by popup_queue::run (and any future backend task that emits to a
       /// WebView window) to avoid the silent-event-drop race when the frontend
       /// listener hasn't yet registered.
       ///
       /// Phase 4 gap closure (04-09): see .planning/debug/webview-warmup-blank-screen.md
       pub async fn wait_for_ready_with_timeout(
           notify: std::sync::Arc<tokio::sync::Notify>,
           timeout: std::time::Duration,
           surface: &'static str,
       ) -> bool {
           match tokio::time::timeout(timeout, notify.notified()).await {
               Ok(()) => {
                   tracing::info!(surface, "WebView ready handshake completed");
                   true
               }
               Err(_) => {
                   tracing::warn!(
                       surface,
                       timeout_ms = timeout.as_millis() as u64,
                       "WebView ready handshake timed out — proceeding anyway"
                   );
                   false
               }
           }
       }
